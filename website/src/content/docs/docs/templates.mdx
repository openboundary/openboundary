## Compliance-Ready Specifications

These templates define the *structure* for compliance-sensitive applications. They include appropriate middleware chains, data isolation patterns, and schema designs.

:::note[Compliance Enforcement]
Generating code from a specification is step one. Full compliance requires audit logging, approval workflows, access governance, and continuous monitoring—capabilities available in [StableBoundary](https://stableboundary.com).
:::

**What compliance requires (beyond code generation):**
- Audit trails for all data access
- Approval workflows for sensitive changes
- Encryption key management
- Access reviews and governance
- Continuous compliance monitoring

## What's in a Template

Each template includes:

| File | Description |
|------|-------------|
| `spec.yaml` | Complete OpenBoundary specification |
| `openapi.yaml` | API contract with request/response schemas |
| `src/db/schema.ts` | Drizzle database schema |
| `src/auth/model.conf` | Casbin authorization model |
| `src/auth/policy.csv` | Role-based access policies |
| `README.md` | Setup and customization guide |

### Example: CRM Template

```
crm/
├── spec.yaml              # 15+ components
├── openapi.yaml           # 40+ endpoints
├── src/
│   ├── auth/
│   │   ├── auth.config.ts # Session-based auth
│   │   ├── model.conf     # RBAC with teams
│   │   └── policy.csv     # Sales, manager, admin roles
│   └── db/
│       └── schema.ts      # contacts, companies, deals, activities
└── README.md
```
